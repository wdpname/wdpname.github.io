第1章 鸟瞰容器生态系统

1.1 容器生态系统

1．容器核心技术

容器核心技术是指能够让Container在host上运行起来的那些技术

（1）容器规范

这两个规范都是由OCI组织发布的。

（2）容器runtime

runtime是容器真正运行的地方。runtime需要跟操作系统kernel紧密协作，为容器提供运行环境。

java程序是容器，那么jvm就是容器runtime。！！

（3）容器管理工具

❓有了容器规范和runtime，为啥要有这个管理工具？

因为就好如，在jvm中启动了一个java程序，那么你可能需要中途停止这个程序，这是需要java命令来中止，java命令就好比管理工具！！！

（4）容器定义工具

❓定义工具是用来干什么的？

在上面提及，容器就相当于java程序，而java程序中有我们用户定义的内容和属性，那么容器中也有自己的内容，那么容器就需要定义工具。

（5）Registry

❓为什么需要一个仓库（Registry）？

因为我们之后需要有许许多多的image，这些镜像需要统一管理，所以用Registry来存储这些镜像。

（6）容器OS

❓为什么需要容器OS？

容器运行需要容器runtime，就好如java程序运行需要jvm，但是jvm实在太大了，有没有更加快速，轻量的运行容器呢？那就要用到容器OS了。

2．容器平台技术

❓为什么要有容器平台技术？

因为容器核心技术能够做到在单个机器上运行容器，但是要是再分布式环境中运行，那么容器核心技术就不满足了。

（1）容器编排引擎

❓什么是容器编排？

类似于多个容器之间如何调度，以及如何服务发现。

（2）容器管理平台

❓什么是容器管理平台？

其实就是在容器编排引擎上实现了一个通用的平台。可以支持多种编排引擎。

（3）基于容器的PaaS

3．容器支持技术

（1）容器网络

❓什么是容器网络？

容器提供互相通信和与外部世界（包括其他容器、主机和互联网）进行通信的网络能力

（2）服务发现

❓什么是服务发现？

因为在微服务环境下，各个服务组件是动态变化的，那么就需要在每次调用一个服务时，就需要服务的发现，因为有些服务会宕机。

（3）监控

❓什么是监控？

其实就是对容器状态的监控。

（4）数据管理

❓为什么需要数据管理？

当容器在各个host之间迁移，如何保证数据也能迁移，这就需要对数据进行管理。

（5）日志管理

❓为什么要有日志管理？

因为一个系统肯定会有一系列的故障，只有做好日志记录，才能对问题进行有逻辑的排查！！！

（6）安全性

❓为什么容器需要具有安全性？

因为不是每一个容器都是安全的，就比如我们去下载软件，不是每一个软件都是良性软件！！！

第3章 Docker镜像

3.1 镜像的内部结构

3.1.1 hello-world——最小的镜像

3.1.2 base镜像

1. rootfs

内核空间是kernel, Linux刚启动时会加载bootfs文件系统，之后bootfs会被卸载掉。用户空间的文件系统是rootfs，包含我们熟悉的 /dev、/proc、/bin等目录。

对于base镜像来说，底层直接用Host的kernel，自己只需要提供rootfs就行了。

2. base镜像提供的是最小安装的Linux发行版

3．支持运行多种Linux OS

不同Linux发行版的区别主要就是rootfs。

所以Docker可以同时支持多种Linux镜像，模拟出多种操作系统环境，如图3-8所示。

3.1.3 镜像的分层结构

❓为什么Docker镜像要采用这种分层结构呢？

因为可以实现共享资源！！！

可写的容器层

当容器启动时，一个新的可写层被加载到镜像的顶部。

这一层通常被称作“容器层”，“容器层”之下的都叫“镜像层”

所有对容器的改动，无论添加、删除，还是修改文件都只会发生在容器层中。只有容器层是可写的，容器层下面的所有镜像层都是只读的。

镜像层数量可能会很多，所有镜像层会联合在一起组成一个统一的文件系统。如果不同层中有一个相同路径的文件，比如 /a，上层的 /a会覆盖下层的 /a，也就是说用户只能访问到上层中的文件 /a。在容器层中，用户看到的是一个叠加之后的文件系统。

只有当需要修改时才复制一份数据，这种特性被称作Copy-on-Write。可见，容器层保存的是镜像变化的部分，不会对镜像本身进行任何修改。

3.2 构建镜像

Docker提供了两种构建镜像的方法： docker commit命令与Dockerfile构建文件。

3.2.1 docker commit

❓既然docker commit不是推荐的方法，我们为什么还要花时间学习呢？

原因是：即便是用Dockerfile（推荐方法）构建镜像，底层也是docker commit一层一层构建新镜像的。学习docker commit能够帮助我们更加深入地理解构建过程和镜像的分层结构。

3.2.2 Dockerfile

Dockerfile是一个文本文件，记录了镜像构建的所有步骤。

3．镜像的缓存特性

Dockerfile中每一个指令都会创建一个镜像层，上层是依赖于下层的

无论什么时候，只要某一层发生变化，其上面所有层的缓存都会失效。

5. Dockerfile常用指令

● FROM指定base镜像。

● MAINTAINER设置镜像的作者，可以是任意字符串。

● COPY将文件从build context复制到镜像。COPY支持两种形式： COPY src dest与COPY [“src”, “dest”]。注意：src只能指定build context中的文件或目录。

● ADD与COPY类似，从build context复制文件到镜像。不同的是，如果src是归档文件（tar、zip、tgz、xz等），文件会被自动解压到dest。

● ENV设置环境变量，环境变量可被后面的指令使用。例如：[插图]

● EXPOSE指定容器中的进程会监听某个端口，Docker可以将该端口暴露出来。我们会在容器网络部分详细讨论。

● VOLUME将文件或目录声明为volume。我们会在容器存储部分详细讨论。

● WORKDIR为后面的RUN、CMD、ENTRYPOINT、ADD或COPY指令设置镜像中的当前工作目录。

● RUN在容器中运行指定的命令。

● CMD容器启动时运行指定的命令。Dockerfile中可以有多个CMD指令，但只有最后一个生效。CMD可以被docker run之后的参数替换。

● ENTRYPOINT设置容器启动时运行的命令。Dockerfile中可以有多个ENTRYPOINT指令，但只有最后一个生效。CMD或docker run之后的参数会被当作参数传递给ENTRYPOINT。

3.3 RUN vs CMD vs ENTRYPOINT

RUN、CMD和ENTRYPOINT这三个Dockerfile指令看上去很类似，很容易混淆。本节将通过实践详细讨论它们的区别。

（1）RUN：执行命令并创建新的镜像层，RUN经常用于安装软件包。

（2）CMD：设置容器启动后默认执行的命令及其参数，但CMD能够被docker run后面跟的命令行参数替换。

（3）ENTRYPOINT：配置容器启动时运行的命令。

3.3.1 Shell和Exec格式

我们可用两种方式指定RUN、CMD和ENTRYPOINT要运行的命令：Shell格式和Exec格式，二者在使用上有细微的区别。

其中使用Exec格式和ENTRYPOINT命令 会有一个缺陷，不能解析系统变量！！！

3.3.4 ENTRYPOINT

ENTRYPOINT看上去与CMD很像，它们都可以指定要执行的命令及其参数。不同的地方在于ENTRYPOINT不会被忽略，一定会被执行，即使运行docker run时指定了其他命令。

ENTRYPOINT有两种格式：

在为ENTRYPOINT选择格式时必须小心，因为这两种格式的效果差别很大。

1. Exec格式

ENTRYPOINT的Exec格式用于设置要执行的命令及其参数，同时可通过CMD提供额外的参数。ENTRYPOINT中的参数始终会被使用，而CMD的额外参数可以在容器启动时动态替换掉。

2. Shell格式

ENTRYPOINT的Shell格式会忽略任何CMD或docker run提供的参数。

综上所述，用exec格式能更加的灵活，可以在docker run 命令后面加上自定义参数。

4.1.3 运行容器的最佳实践

按用途容器大致可分为两类：服务类容器和工具类的容器。

服务类容器以daemon的形式运行，对外提供服务，比如Web Server、数据库等。通过 -d以后台方式启动这类容器是非常合适的。如果要排查问题，可以通过exec -it进入容器。

工具类容器通常能给我们提供一个临时的工作环境，通常以run -it方式运行，比如如图4-14所示的例子。

运行busybox, run -it的作用是在容器启动后就直接进入。我们这里通过wget验证了在容器中访问internet的能力。执行exit退出终端，同时容器停止。

工具类容器多使用基础镜像，例如busybox、debian、ubuntu等。

4.2 stop/start/restart容器

容器在docker host中实际上是一个进程，docker stop命令本质上是向该进程发送一个SIGTERM信号。如果想快速停止容器，可使用docker kill命令，其作用是向容器进程发送SIGKILL信号，如图4-16所示。

对于处于停止状态的容器，可以通过docker start重新启动，如图4-17所示。

docker start会保留容器的第一次启动时的所有参数。

docker restart可以重启容器，其作用就是依次执行docker stop和docker start。

容器可能会因某种错误而停止运行。对于服务类容器，我们通常希望在这种情况下容器能够自动重启。启动容器时设置 —restart就可以达到这个效果，如图4-18所示。

—restart=on-failure:3，意思是如果启动进程退出代码非0，则重启容器，最多重启3次。

4.3 pause / unpause容器

有时我们只是希望让容器暂停工作一段时间，比如要对容器的文件系统打个快照，或者dcoker host需要使用CPU，这时可以执行docker pause，如图4-19所示。
